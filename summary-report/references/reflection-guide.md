# 深度反思指引

生成“深度反思”时，优先依据可观察信号，不要只给抽象评价。

## 1. 效能评估

优先找以下信号：

- 一次工具调用直接完成目标
- 多个独立读取/检查动作被并行处理
- 复用了现有文件、模式或上下文，没有重复造轮子
- 先澄清了关键约束，减少返工

低效信号：

- 同一信息被重复读取或重复总结
- 明明可以批量操作，却拆成冗长串行步骤
- 生成了用户没要求的额外产物
- 因默认假设错误导致返工

## 2. 逻辑漏洞

重点检查：

- 是否误解了用户目标、范围、语言偏好或输出形式
- 是否引用了未定义变量、路径、工具或环境前提
- 是否要求执行环境具备其实并不存在的能力
- 是否把“建议”写成了“必须”，导致流程脆弱

常见例子：

- 把“优化一个 skill”执行成“新建一个 skill”
- 默认存在 `source_path`、Slack `message`、外部脚本、子技能
- 未考虑文件不存在、目录未创建、权限不足

## 3. 知识盲区

优先识别三类盲区：

- 工具盲区：不了解工具可用性、参数语义或权限边界
- 领域盲区：不了解用户项目中的约定、目录结构、命名习惯
- 推理盲区：中间状态丢失、因信息不足却做了强结论

写法要求：

- 说明“盲区是什么”
- 说明“为什么会卡住”
- 说明“下次怎么更早发现”

## 4. 记忆与策略优化

尽量把教训改写成下面这种形式：

```text
【准则】当遇到 {触发条件} 时，应 {具体行动}，而非 {错误做法}。
原因：{为什么这样更稳}
```

示例：

```text
【准则】当 skill 指令中出现未定义变量时，应先把变量来源改成显式输入或降级为可缺省字段，而非默认环境里一定存在该变量。
原因：隐式依赖会让 skill 在不同执行环境下直接失效。
```

## 5. 进化方案排序

按以下优先级生成改进动作：

1. P0 正确性：修复会直接导致错误输出或无法执行的问题
2. P1 稳定性：移除隐式依赖、补充缺省路径和失败分支
3. P2 效率：减少冗余步骤、减少重复读取与重复总结
4. P3 体验：改善结果表达、交付方式和可读性
