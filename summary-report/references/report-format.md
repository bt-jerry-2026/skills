# 每日复盘报告格式规范

## 使用规则

- 输出必须为 Markdown
- 所有一级、二级标题名称保持一致，便于后续程序解析
- 不知道的字段可写 `N/A`
- 若数据源不止一个，用中文逗号列出，不要编造单一 `source_path`
- “当日轨迹”只写真实发生过的任务

## 报告模板

```markdown
# 每日复盘报告 — YYYY-MM-DD

## [Meta Info]

| 字段 | 值 |
|------|-----|
| 复盘时间 | {本地时间，精确到分钟} |
| 会话任务数 | {N} |
| 数据源 | {当前会话上下文 / 路径列表 / N/A} |
| 模型版本 | {model_id / N/A} |

---

## [当日轨迹]

> 按时间或执行顺序概括今天真实发生的关键任务。

| # | 任务描述 | 状态 | 耗时评估 | 证据来源 |
|---|---------|------|---------|---------|
| 1 | {任务简述} | ✅ / ⚠️ / ❌ | 短 / 中 / 长 | {会话 / memory / 日志} |

---

## [深度反思]

### 效能评估
{哪些动作高效，为什么；哪些步骤冗余，冗余在哪里}

### 逻辑漏洞
{是否出现意图理解偏差、执行路径错误、工具选择不当；尽量给出具体信号}

### 知识盲区
{哪些问题无法直接解决、需要试错、或依赖外部知识；说明原因}

### 记忆与策略优化
{将教训写成下次可复用的规则；优先写“触发条件 + 行动”}

---

## [进化方案]

> 只写后续真正可执行、可验证的改进动作，按优先级排序。

1. **{优化项}**：{触发条件}；{具体动作}；{预期效果}
2. **{优化项}**：{触发条件}；{具体动作}；{预期效果}
3. **{优化项}**：{触发条件}；{具体动作}；{预期效果}

---

## [自我寄语]

> 给明天的自己一句最具体的执行提醒。

{一句直接、可执行的话}
```

## 质量标准

- “当日轨迹”与“深度反思”之间要能互相对应
- “进化方案”必须能落回当天暴露的问题
- 不要出现“整体良好”“继续努力”这类空话
